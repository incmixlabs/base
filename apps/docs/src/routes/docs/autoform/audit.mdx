<Heading as="h2" size="lg" id="covered-slice" data-heading>
  What Is Already Covered
</Heading>

<Text size="sm" className="mt-2 text-muted-foreground">
  The current AutoForm slice is strong enough for a first real E2E pass. It already covers:
</Text>

<ul>
  <li>native normalized renderer</li>
  <li>AJV validation and field error rendering</li>
  <li>submit and pending-submit lifecycle</li>
  <li>runtime conditions on existing fields</li>
  <li>branch-only conditional materialization at root, nested-object, and repeater-item scopes</li>
  <li>post-submit field-level and form-level server error handling</li>
  <li>array repeater add/remove/reorder behavior</li>
  <li>responsive layout behavior</li>
  <li>stepper-based multi-step flows</li>
  <li>Playwright browser coverage for the completed lifecycle slice</li>
</ul>

<Heading as="h2" size="lg" id="gaps" data-heading>
  Remaining Gaps
</Heading>

<Text size="sm" className="mt-2 text-muted-foreground">
  The most important open gaps are architectural, not cosmetic:
</Text>

<ul>
  <li>arrays still need stronger array-level UX and richer item lifecycle operations beyond add/remove/reorder</li>
  <li>wrapper parity still drops sections, steps, native condition runtime, and richer widget behavior</li>
  <li>rich widget props are still not a complete first-class normalized contract across all surfaces</li>
</ul>

<Heading as="h2" size="lg" id="next-order" data-heading>
  Recommended Next Order
</Heading>

<ol>
  <li>Deepen wrapper parity beyond the adapter baseline, starting with higher-fidelity dialog behavior and richer widget/runtime support.</li>
  <li>Continue improving array-level UX around repeaters, especially beyond the current add/remove/reorder lifecycle.</li>
  <li>Add first-class date validation rules and renderer constraints.</li>
</ol>
